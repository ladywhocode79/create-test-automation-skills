## Overview

This document captures every design decision and architectural rationale for the mock
infrastructure skill. It is the canonical reference for extending, debugging, or replacing
the WireMock setup generated by this skill.

The mock skill is **language-agnostic**. It generates WireMock infrastructure that works
identically with both the Python and Java language tracks. The Strategy pattern wiring
(client factory, real/mock switching) is generated by the language track skill; this skill
generates the WireMock-specific infrastructure on top.

**What this skill generates:**

```
mocks/
├── stubs/
│   ├── user_stubs.json       ← WireMock mappings for User resource
│   └── auth_stubs.json       ← WireMock mappings for auth/token endpoint
├── files/                    ← (optional) static response files
└── wiremock_lifecycle.py     ← Python: register_stub(), verify_called(), reset_stubs()
    WireMockLifecycle.java    ← Java: registerStub(), verifyCalled(), resetStubs()

docker-compose.yml            ← WireMock Docker service (added/merged with existing)
```

### Block 1 — Invocation Context

**Purpose:** Understand when and how this skill is invoked by the orchestrator.

**Trigger:** Orchestrator Q6 — "Mock mode?"

```
[A] Real only       → mock skill NOT invoked
[B] Mock only       → mock skill invoked; TEST_MODE=mock hardcoded in .env.example
[C] Real + Mock     → mock skill invoked; ClientFactory generates both branches
```

**Input from orchestrator:**

**Files this skill generates vs. language track:**
- **This skill:** `docker-compose.yml`, `mocks/stubs/*.json`, `mocks/wiremock_lifecycle.*`

- Both run in the same Phase 2 scaffold generation — orchestrator coordinates the order

### Block 2 — WireMock Docker Setup

**Purpose:** Run WireMock as a Docker container so it is language-agnostic, version-pinned,
and starts/stops without affecting the local machine's port state.

**Generated file:** `docker-compose.yml`

```yaml
services:
  wiremock:
    image: wiremock/wiremock:3.3.1
    container_name: wiremock
    ports:
      - "8080:8080"
    volumes:
      - ./mocks/stubs:/home/wiremock/mappings    # stubs auto-loaded at startup
      - ./mocks/files:/home/wiremock/__files      # static response body files (optional)
    command: >
      --global-response-templating
      --verbose
      --port 8080
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/__admin/health"]
      interval: 5s
      timeout: 3s
      retries: 10
      start_period: 5s
```

**Local usage:**

```bash
docker-compose up -d wiremock          # start in background
docker-compose logs -f wiremock        # tail logs
docker-compose down                    # stop + remove container

curl http://localhost:8080/__admin/health      # verify running
curl http://localhost:8080/__admin/mappings    # list loaded stubs
```

**Volume mount behavior:**
The `./mocks/stubs` directory is mounted into WireMock's `/home/wiremock/mappings`.
WireMock loads all `.json` files from that directory at startup. Adding a new stub file
requires only a container restart — no WireMock configuration changes.

**Healthcheck gate:**
The healthcheck prevents tests from starting before WireMock is ready. In CI, use
Docker's `service_healthy` condition to gate the test step. Locally, `docker-compose up -d`
returns before healthcheck passes — always wait for `/__admin/health` to return 200 before running tests.

### Block 3 — Stub Files: User Resource

**Purpose:** Canned HTTP responses for all User API endpoints. Stubs are JSON mappings
that WireMock reads at startup from the mounted volume.

**Generated file:** `mocks/stubs/user_stubs.json`

**Stub anatomy:**

```json
{
  "mappings": [
    {
      "name": "post-user-success",              // required — for debuggability
      "request": {
        "method": "POST",
        "url": "/api/v1/users",
        "headers": {
          "Content-Type": { "equalTo": "application/json" },
          "Authorization": { "matches": "Bearer .+" }   // regex match
        }
      },
      "response": {
        "status": 201,
        "headers": { "Content-Type": "application/json" },
        "jsonBody": {
          "id": 1001,
          "username": "{{jsonPath request.body '$.username'}}",   // echo from request
          "email":    "{{jsonPath request.body '$.email'}}",
          "role":     "{{jsonPath request.body '$.role'}}",
          "created_at": "{{now format='yyyy-MM-dd\\'T\\'HH:mm:ss\\'Z\\''}}"
        }
      }
    }
  ]
}
```

**Stubs generated for User resource (all modes):**

| Stub name                  | Method | URL pattern               | Status | Notes                          |
|----------------------------|--------|---------------------------|--------|--------------------------------|
| `post-user-success`        | POST   | `/api/v1/users`           | 201    | Echoes username/email/role     |
| `get-user-by-id-success`   | GET    | `/api/v1/users/[1-9][0-9]*` | 200  | Returns fixed user object      |
| `get-user-not-found`       | GET    | `/api/v1/users/9{4,}`     | 404    | Priority 1 (overrides general) |
| `get-user-list-success`    | GET    | `/api/v1/users`           | 200    | Returns 2-item list            |
| `delete-user-success`      | DELETE | `/api/v1/users/[1-9][0-9]*` | 204  | No response body               |

**How the `9999` not-found pattern works:**
The `"urlPattern": "/api/v1/users/9{4,}"` matches IDs like `9999`, `99999`.
Test code uses `999999L` as the "non-existent" ID by convention.
`priority: 1` ensures this stub matches before the general "any numeric ID" stub.

### Block 4 — Stub Files: Auth Resource

**Purpose:** Canned responses for the token endpoint. Generated only when `auth_type != none`.

**Generated file:** `mocks/stubs/auth_stubs.json`

```json
{
  "mappings": [
    {
      "name": "post-token-client-credentials-success",
      "request": {
        "method": "POST",
        "url": "/oauth/token",
        "bodyPatterns": [
          { "contains": "grant_type=client_credentials" }
        ]
      },
      "response": {
        "status": 200,
        "headers": { "Content-Type": "application/json" },
        "jsonBody": {
          "access_token": "mock-access-token-abc123xyz",
          "token_type": "Bearer",
          "expires_in": 3600,
          "scope": "read write"
        }
      }
    },
    {
      "name": "post-token-invalid-credentials",
      "priority": 1,
      "request": {
        "method": "POST",
        "url": "/oauth/token",
        "bodyPatterns": [
          { "contains": "client_id=invalid" }
        ]
      },
      "response": {
        "status": 401,
        "headers": { "Content-Type": "application/json" },
        "jsonBody": {
          "error": "invalid_client",
          "error_description": "Client authentication failed"
        }
      }
    }
  ]
}
```

**Why a fixed `mock-access-token-abc123xyz`:**
The mock access token is a known constant. Tests verify the Authorization header is sent;
WireMock validates `"Authorization": { "matches": "Bearer .+" }` — any Bearer token passes.
The token value is irrelevant in mock mode since WireMock doesn't validate JWT signatures.

**`TOKEN_URL` in mock mode:**
Set `TOKEN_URL=http://localhost:8080/oauth/token` in `.env.example` for mock mode.
`AuthManager` fetches from WireMock, which returns the canned token. No real auth server needed.

### Block 5 — Response Templating

**Purpose:** Make stub responses dynamic — echo request fields back, generate timestamps,
create random IDs. Enabled by `--global-response-templating` in docker-compose command.

**Available helpers:**

```
{{jsonPath request.body '$.fieldName'}}         Extract field from JSON request body
{{request.headers.Authorization}}               Access any request header value
{{request.pathSegments.[2]}}                    URL path segment (0-indexed)
{{randomValue length=8 type='ALPHANUMERIC'}}    Random alphanumeric string
{{randomValue length=6 type='NUMERIC'}}         Random numeric string
{{now format='yyyy-MM-dd'}}                     Current date
{{now format='yyyy-MM-dd\'T\'HH:mm:ss\'Z\''}}  ISO 8601 datetime
{{now offset='3 days' format='yyyy-MM-dd'}}     Offset from today
```

**Key use case — echo request body in response:**
The `post-user-success` stub echoes `username`, `email`, and `role` from the request body
back in the response. This allows `assertThat(user.username()).isEqualTo(payload.username())`
to pass without the stub needing to know what username the factory generated.
Without templating, the stub would return a hardcoded username that often doesn't match
the test's assertion.

**Why `--global-response-templating` (not per-stub):**
The alternative is adding `"transformers": ["response-template"]` to every stub that uses
templating. Global mode enables templating on all stubs — simpler to manage, no risk of
forgetting the transformer on a new stub. The only downside (literal `{{` in responses)
is avoidable by escaping: `{{{{` renders as `{{`.

### Block 6 — Stub Naming and Priority

**Purpose:** Every stub has a `"name"` field and some have a `"priority"` field.
These two fields make stub debugging and conflict resolution deterministic.

**Naming convention:**

```
{http-method}-{resource}-{scenario}
```

| Example name                      | Method | Resource | Scenario            |
|-----------------------------------|--------|----------|---------------------|
| `post-user-success`               | POST   | user     | success             |
| `get-user-not-found`              | GET    | user     | not found           |
| `post-token-client-credentials-success` | POST | token | CC grant success  |
| `get-user-list-empty`             | GET    | user     | empty list          |
| `delete-user-forbidden`           | DELETE | user     | 403 forbidden       |

**Why `"name"` is required:**
WireMock's `/__admin/requests` log shows matched stub names. Without names, the log
shows UUID identifiers that are meaningless during debugging. With names, you can
immediately see "ah, the test matched `get-user-not-found` when I expected `get-user-by-id-success`".

**Priority system:**
Lower number = higher priority. Default priority when unset = 5.

```json
// General stub: matches any numeric user ID
{
  "name": "get-user-by-id-success",
  "request": { "method": "GET", "urlPattern": "/api/v1/users/([1-9][0-9]*)" },
  ...
}

// Specific stub: matches only IDs starting with 9999 — overrides general stub
{
  "name": "get-user-not-found",
  "priority": 1,
  "request": { "method": "GET", "urlPattern": "/api/v1/users/9{4,}" },
  ...
}
```

WireMock evaluates all matching stubs and picks the one with the lowest priority number.
Without priority, WireMock picks the most recently registered matching stub — unpredictable
when multiple stubs could match the same request.

### Block 7 — WireMock Admin API

**Purpose:** Programmatic access to WireMock at runtime. Used by `wiremock_lifecycle` helpers
for per-test stub overrides, call verification, and state reset.

**Full admin API reference:**

| Operation              | Method | Endpoint                     | Use case                        |
|------------------------|--------|------------------------------|---------------------------------|
| Register stub          | POST   | `/__admin/mappings`          | Per-test dynamic stubs          |
| List all stubs         | GET    | `/__admin/mappings`          | Debugging: see what's loaded    |
| Reset runtime stubs    | POST   | `/__admin/mappings/reset`    | After per-test stub teardown    |
| Get request log        | GET    | `/__admin/requests`          | Debug: what calls were made     |
| Count requests         | POST   | `/__admin/requests/count`    | Verify call count assertions    |
| Find requests          | POST   | `/__admin/requests/find`     | Verify specific request details |
| Reset request log      | DELETE | `/__admin/requests`          | Before each test (clean state)  |
| Health check           | GET    | `/__admin/health`            | Startup gating                  |

**Important distinction — two types of stubs:**
1. **File-based stubs** (`mocks/stubs/*.json`): Loaded at WireMock startup. Persistent across tests.
   Best for: happy-path stubs that most tests use.
2. **Runtime stubs** (via `POST /__admin/mappings`): Registered during a test. Reset via `POST /__admin/mappings/reset`.
   Best for: scenario-specific stubs (500 errors, edge cases, per-test overrides).

### Block 8 — WireMock Lifecycle Helpers

**Purpose:** Thin wrapper around the WireMock admin API. Tests use these helpers instead
of raw `requests.post("/__admin/mappings", ...)` calls.

**Generated file (Python):** `mocks/wiremock_lifecycle.py`

```python
import requests

WIREMOCK_URL = "http://localhost:8080"

def register_stub(stub: dict) -> None:
    """Register a runtime stub via WireMock admin API."""
    response = requests.post(f"{WIREMOCK_URL}/__admin/mappings", json=stub)
    response.raise_for_status()

def reset_stubs() -> None:
    """Remove all runtime stubs (file-based stubs remain)."""
    requests.post(f"{WIREMOCK_URL}/__admin/mappings/reset").raise_for_status()

def reset_requests() -> None:
    """Clear the request journal (call history)."""
    requests.delete(f"{WIREMOCK_URL}/__admin/requests").raise_for_status()

def verify_called(url_pattern: str, times: int = 1, method: str = "GET") -> None:
    """Assert that WireMock received exactly N requests matching the pattern."""
    payload = {
        "method": method,
        "urlPattern": url_pattern
    }
    response = requests.post(f"{WIREMOCK_URL}/__admin/requests/count", json=payload)
    response.raise_for_status()
    actual = response.json()["count"]
    assert actual == times, (
        f"Expected {times} call(s) to {method} {url_pattern}, got {actual}"
    )
```

**Generated file (Java):** `src/main/java/{pkg}/mocks/WireMockLifecycle.java`

```java
public class WireMockLifecycle {
    private static final String WIREMOCK_URL = "http://localhost:8080/__admin";

    public static void registerStub(Map<String, Object> stub) {
        // POST /__admin/mappings with stub as JSON body
    }

    public static void resetStubs() {
        // POST /__admin/mappings/reset
    }

    public static void resetRequests() {
        // DELETE /__admin/requests
    }

    public static void verifyCalled(String urlPattern, int times, String method) {
        // POST /__admin/requests/count — assert count == times
    }
}
```

**Per-test stub override pattern (Python):**

```python
# conftest.py — fixture for tests that need per-test stub control
@pytest.fixture
def wiremock():
    reset_requests()       # clear call history before test
    yield                  # test runs
    reset_stubs()          # remove any runtime stubs registered during test

# In a test:
def test_api_returns_500_is_handled_gracefully(api_client, wiremock):
    register_stub({
        "name": "get-user-list-server-error",
        "request": {"method": "GET", "url": "/api/v1/users"},
        "response": {"status": 503, "jsonBody": {"error": "Service unavailable"}}
    })

    response = api_client.get("/api/v1/users")
    assert response.status_code == 503
```

**`autouse=False` by default:**
The `wiremock` fixture is opt-in. Tests that use only file-based stubs don't need it.
Tests that register runtime stubs must declare `wiremock` as a parameter — this documents
the dependency explicitly and ensures cleanup.

### Block 9 — UI Backend Mocking

**Purpose:** For UI (browser) tests, WireMock replaces the backend API that the application
calls — not the browser itself. Tests interact only with page objects; WireMock is transparent.

**Architecture:**

```
[TestNG/Pytest Test]
       ↓
[Browser — Playwright or Selenium]
       ↓
[Your Application (AUT) — started with API_BASE_URL=http://localhost:8080]
       ↓
[WireMock — mocks the real backend API]
```

**Environment setup for UI + mock:**

```bash
# .env — local
AUT_BASE_URL=http://localhost:3000          # your app's frontend
BACKEND_URL=http://localhost:8080           # WireMock replaces real API
TEST_MODE=mock
```

**Starting the application pointing at WireMock:**

```bash
# React/Next.js:
REACT_APP_API_URL=http://localhost:8080 npm start

# Generic:
API_BASE_URL=http://localhost:8080 ./start-app.sh
```

The application must support a configurable backend URL. This is standard for any
well-architected frontend. If the application has a hardcoded backend URL, WireMock
cannot intercept its calls without a network proxy (out of scope for this skill).

**Docker Compose — full UI stack:**

```yaml
services:
  wiremock:
    image: wiremock/wiremock:3.3.1
    ports: ["8080:8080"]
    volumes:
      - ./mocks/stubs:/home/wiremock/mappings
    command: --global-response-templating --verbose
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/__admin/health"]
      interval: 5s
      retries: 10

  app:
    image: your-app:latest
    ports: ["3000:3000"]
    environment:
      API_BASE_URL: http://wiremock:8080    # Docker network — app → wiremock
    depends_on:
      wiremock:
        condition: service_healthy          # wait for WireMock before starting app
```

**Cross-layer verification — UI test + WireMock call count:**

```python
def test_login_calls_auth_endpoint(page, wiremock):
    login_page = LoginPage(page).navigate()
    login_page.login_as(LoginFormData.build("user@example.com", "password123"))

    # UI assertion: user is on dashboard
    assert DashboardPage(page).is_loaded()

    # Backend assertion: app called token endpoint exactly once
    verify_called(url_pattern="/oauth/token", times=1, method="POST")
```

This cross-layer assertion catches bugs where the UI navigates correctly but the
application skips the auth call (e.g., using a cached token unexpectedly).

### Block 10 — Strategy Pattern: Real/Mock Switching

**Purpose:** Switch between real and mock backends by changing one env var. Zero test code changes.

**Concept (language-agnostic):**

```
«interface / base class»
ApiClient
    + get(endpoint) → Response
    + post(endpoint, body) → Response
    + put(endpoint, body) → Response
    + delete(endpoint) → Response
            ▲
            │ implements
    ┌───────┴────────┐
    │                │
RealApiClient    MockApiClient
(hits BASE_URL)  (hits WIREMOCK_BASE_URL = localhost:8080)
            ▲
            │ creates
     ClientFactory.build(TEST_MODE)
```

**Python implementation (generated by python track skill):**

```python
def build_client(token=None):
    mode = os.getenv("TEST_MODE", "mock").lower()
    match mode:
        case "mock": return MockApiClient(token=token)
        case "real": return RealApiClient(token=token)
        case _: raise ValueError(f"Unsupported TEST_MODE: {mode}")
```

**Java implementation (generated by java track skill):**

```java
public class ClientFactory {
    public static BaseApiClient build(String token) {
        return switch (EnvConfig.get().testMode().toLowerCase()) {
            case "mock" -> new MockApiClient(token);
            case "real" -> new BaseApiClient(token);
            default -> throw new IllegalArgumentException("Unsupported TEST_MODE");
        };
    }
}
```

**Runtime switching — no code changes:**

```bash
# Fast, isolated regression — all tests against WireMock:
TEST_MODE=mock pytest -m "regression" -n auto

# Integration confidence — smoke tests against real API:
TEST_MODE=real pytest -m "smoke"

# CI: parallel jobs
# job 1: TEST_MODE=mock pytest -m "regression"
# job 2: TEST_MODE=real pytest -m "smoke"
```

**MockApiClient — only difference from BaseApiClient:**

```python
# Python
class MockApiClient(BaseApiClient):
    def __init__(self, token=None):
        super().__init__(token=token)
        self.base_url = "http://localhost:8080"   # WireMock port

# Java
public class MockApiClient extends BaseApiClient {
    // Override base URI to WireMock at construction
    // Spec built with EnvConfig.get().wireMockBaseUrl() instead of baseUrl()
}
```

## Design Decisions

---

### D1 — WireMock as Default Mock Tool

**Options considered:**

| Option | Tool      | Approach                                       | Trade-offs                                              |
|--------|-----------|------------------------------------------------|---------------------------------------------------------|
| A      | WireMock  | Docker-based, JSON stub files, admin REST API  | Manual stub authoring. No OpenAPI import.              |
| B      | Prism     | OpenAPI spec → auto-generates stubs            | Requires OpenAPI spec. Auto-stubs may be too permissive.|
| C      | Mockoon   | GUI-based tool with import/export              | Not developer-friendly for CI. GUI-first workflow.     |

**Chosen:** Option A — WireMock.

**Why:** WireMock is the most widely used API mock server in enterprise test automation.
Its JSON stub format is expressive, version-controllable, and IDE-readable. The Docker image
is stable and well-maintained. Response templating covers most dynamic response needs.
The admin REST API enables per-test stub overrides without a library dependency.

**Trade-off (documented in orchestrator D3):** WireMock has no built-in OpenAPI spec import.
Every stub must be hand-authored. Teams with an OpenAPI/Swagger spec should consider Prism
instead — Prism auto-generates responses from the spec with one command: `prism mock openapi.yaml`.
The Strategy pattern makes switching to Prism trivial: add `PrismApiClient` (5 lines) and
one branch in `ClientFactory` (2 lines). No test code changes needed.

**When to recommend Prism over WireMock:**
- User mentions "OpenAPI spec", "Swagger file", or "spec-first development" at Q7
- The API is still under development and the spec changes frequently
- The team wants validation that requests conform to the OpenAPI spec

---

### D2 — Docker-Based WireMock (not Embedded)

**Options considered:**

| Option | Approach                  | Trade-offs                                                   |
|--------|---------------------------|--------------------------------------------------------------|
| A      | Docker container          | Language-agnostic, version-pinned, CI-native.                |
| B      | Embedded WireMock (Java)  | In-process, no Docker required. Java-only.                   |
| C      | WireMock standalone JAR   | No Docker. Requires Java on all machines. Version management. |

**Chosen:** Option A — Docker container.

**Why:** Docker is language-agnostic — the same `docker-compose.yml` works for the Python
and Java tracks without change. The WireMock version is pinned in the image tag (`3.3.1`),
preventing unexpected behavior from version drift. CI services (GitHub Actions, GitLab CI)
support Docker service containers natively, making the same `docker-compose.yml` reusable
in both local development and CI.

**Trade-off:** Requires Docker installed locally. In environments without Docker, the
embedded WireMock option (Java only: `wireMockServer.start()`) is the alternative.
The skill could add a Java-only embedded mode in a future version as `TEST_MOCK_MODE=embedded`.

---

### D3 — `--global-response-templating` Enabled by Default

**Options considered:**

| Option | Approach                          | Trade-offs                                              |
|--------|-----------------------------------|---------------------------------------------------------|
| A      | Global flag in docker command     | All stubs can use templates. No per-stub transformer.   |
| B      | Per-stub `"transformers"` field   | Explicit opt-in per stub. More verbose.                 |
| C      | No templating                     | Simpler stubs. Hardcoded responses can't echo request.  |

**Chosen:** Option A — global flag.

**Why:** The most critical templating use case is echoing request fields in responses
(e.g., `{{jsonPath request.body '$.username'}}`). Without this, `POST /users` returns
a hardcoded username that breaks `assertThat(user.username()).isEqualTo(payload.username())`.
Global mode enables templating on every stub — no risk of forgetting the per-stub `"transformers"`
field when authoring a new stub.

**Downside of global templating:** If a stub response legitimately needs to contain `{{`
as literal text, it must be escaped as `{{{{`. This is rare in API responses.

### D4 — `TEST_MODE` Env Var for Strategy Selection

**Options considered:**

| Option | Approach                        | Trade-offs                                               |
|--------|---------------------------------|----------------------------------------------------------|
| A      | `TEST_MODE` environment variable | Zero test code changes. CI-native. Works everywhere.   |
| B      | Test annotation / marker        | Explicit per-test. Requires test modification to switch. |
| C      | Config file / profile           | Structured. Requires profile management tooling.        |

**Chosen:** Option A — environment variable.

**Why:** Environment variables are the universal CI/CD configuration mechanism.
Switching `TEST_MODE=mock` in a GitHub Actions step requires no code change — the same
test suite runs against WireMock or the real API just by changing one env var.
Annotations (Option B) require modifying test files to change scope — every test that
should switch needs updating. Config profiles (Option C) add a build tool dependency.

### D5 — File-Based Stubs for Static Cases, Runtime Stubs for Per-Test

**Options considered:**

| Option | Approach                     | Trade-offs                                                  |
|--------|------------------------------|-------------------------------------------------------------|
| A      | File-based only              | Simple. No admin API needed. Can't do per-test overrides.   |
| B      | Runtime-only (all via API)   | Flexible. Requires setup in every test. No persistence.     |
| C      | File-based + runtime (hybrid)| Happy-path in files. Edge cases via API. Clean separation.  |

**Chosen:** Option C — hybrid.

**Why:** Happy-path stubs (200 create, 200 get, 204 delete) are used by most tests.
Duplicating them via admin API in every test `@BeforeMethod` is wasteful and noisy.
File-based stubs load once at WireMock startup and persist until the container stops.
Runtime stubs handle test-specific scenarios: 500 errors, empty lists, edge cases.
After each test, `reset_stubs()` removes only runtime stubs — file-based stubs remain.
This gives clean test isolation without re-registering base stubs for every test.

### D6 — Stub Naming Convention (`"name"` Field Required)

**Options considered:**

| Option | Approach            | Trade-offs                                                |
|--------|---------------------|-----------------------------------------------------------|
| A      | Required `"name"` in all stubs | Debugging-first. Extra 3 characters per stub. |
| B      | `"name"` optional   | Less boilerplate. WireMock uses UUID. Unreadable in logs.  |

**Chosen:** Option A — required `"name"` field in all stubs.

**Why:** WireMock's `/__admin/requests` log identifies matched stubs by name (if set)
or by UUID (if not). UUID `"id":"3a7c..."` in the log is meaningless during debugging.
Name `"post-user-success"` immediately identifies which stub matched. The convention
`{method}-{resource}-{scenario}` is consistent and searchable across all stub files.

---

### D7 — Priority System for Stub Resolution

**Options considered:**

| Option | Approach                    | Trade-offs                                             |
|--------|-----------------------------|--------------------------------------------------------|
| A      | `"priority"` field (lower = higher priority) | Explicit, deterministic. |
| B      | Most-specific URL wins       | Implicit. WireMock's default behavior is ambiguous.    |
| C      | Registration order           | Last-registered wins. Unpredictable with parallel tests.|

**Chosen:** Option A — explicit `"priority"` field on override stubs.

**Why:** WireMock's default stub selection when multiple stubs match is not documented
as stable. Relying on registration order (Option C) breaks with parallel test execution.
Explicit priority makes the resolution order clear in the stub file itself — a new team
member can read `"priority": 1` and understand this stub takes precedence. Priority is
only set on stubs that intentionally override a more general stub; most stubs have no
priority field (defaulting to 5).

## Known Limitations

1. **No OpenAPI spec import** — WireMock stubs must be hand-authored. The `user_stubs.json`
   generated by this skill is a complete example for the User resource domain. Additional
   resource domains (Orders, Products, etc.) require creating new stub files following the same
   pattern. See D1 for Prism as an alternative when an OpenAPI spec exists.

2. **WireMock does not validate request body schema** — it matches on patterns
   (`bodyPatterns.contains`, `bodyPatterns.matchesJsonPath`) but does not validate that the
   full JSON body conforms to a schema. Schema validation in tests is performed by the
   language track skill's Jackson (Java) or Pydantic (Python) deserialization.

3. **Stateful scenarios not supported by default** — WireMock supports stateful stubs via
   Scenarios (state machine), but they are not generated by this skill. If a test sequence
   requires "create user → verify user appears in list → delete user → verify gone", each
   step needs a separate stub registered in order. Consider using `TEST_MODE=real` for
   stateful end-to-end flows.

4. **Application must support configurable backend URL** — UI backend mocking requires
   the application to accept its backend API URL as a configuration parameter at startup.
   Applications with hardcoded API URLs cannot be pointed at WireMock without a network-level
   proxy (out of scope).

5. **Docker required** — the Docker-based approach requires Docker Desktop (local) or a
   Docker daemon (CI). In environments without Docker, fall back to the WireMock standalone
   JAR (Java only: `java -jar wiremock-standalone.jar`) or embedded WireMock for Java.
