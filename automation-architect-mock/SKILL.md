# Mock Service Skill — automation-architect

This skill is invoked by the automation-architect orchestrator after the
user selects a mock mode (B or C). It generates the mock infrastructure
files on top of the language track scaffold.

## Files Generated by This Skill

```
mocks/
├── stubs/
│   ├── user_stubs.json         ← WireMock mappings for User resource
│   └── auth_stubs.json         ← WireMock mappings for auth/token endpoint
└── wiremock_lifecycle.{ext}    ← register_stub(), verify_called(), reset()

docker-compose.yml              ← WireMock Docker service (added/merged)
```

Strategy pattern files are generated within the track skill's Layer 2
output (see language track references). This skill generates the
WireMock-specific infrastructure only.

## Delegation Rules

1. Load `references/wiremock-docker-setup.md` always
2. Load `references/wiremock-ui-backend-mocking.md` if test_type includes UI
3. Load `references/strategy-pattern-switch.md` for Strategy pattern guidance
   (actual code is in language track Layer 2 references)
4. Load `references/prism-setup.md` only if user selected Prism
5. Load `references/mockoon-setup.md` only if user selected Mockoon

## WireMock Admin API Reference

WireMock exposes an admin REST API at `/__admin` on the same port:

| Operation | Method | Endpoint |
|---|---|---|
| Register stub | POST | `/__admin/mappings` |
| List all stubs | GET | `/__admin/mappings` |
| Reset runtime stubs | POST | `/__admin/mappings/reset` |
| Get request log | GET | `/__admin/requests` |
| Count requests | POST | `/__admin/requests/count` |
| Find requests | POST | `/__admin/requests/find` |
| Reset requests log | DELETE | `/__admin/requests` |
| Health check | GET | `/__admin/health` |

## Stub Naming Convention

Every stub MUST have a `"name"` field for debuggability:

```json
{
  "name": "{http-method}-{resource}-{scenario}",
  "request": { ... },
  "response": { ... }
}
```

Examples:
- `"post-user-success"`
- `"get-user-not-found"`
- `"post-token-valid-credentials"`
- `"get-user-list-empty"`

## Stub Priority

When multiple stubs match the same request, WireMock uses the `priority`
field (lower number = higher priority):

```json
{
  "name": "get-user-9999-not-found",
  "priority": 1,
  "request": { "method": "GET", "url": "/api/v1/users/9999" },
  "response": { "status": 404, ... }
}
```

Use priority stubs for specific test scenarios that need to override
the general stub.
