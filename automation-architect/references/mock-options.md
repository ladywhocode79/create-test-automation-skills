# Mock Service Options

This file defines available mock tools, selection logic, and how WireMock
(the default) integrates with both API and UI test profiles.

---

## Tool Comparison

```
Tool          Type              Language    Best For                       Docker?
────────────────────────────────────────────────────────────────────────────────
WireMock      Standalone HTTP   Any         Team-shared stubs, API + UI    Yes
              stub server                   backend mocking, rich matching
Prism         OpenAPI-driven    Any         Auto-stubs from swagger.json,  Yes (or CLI)
              mock server                   contract validation, zero
                                            stub maintenance
Mockoon       GUI + CLI         Any         Easy onboarding, stubs as      No (app/CLI)
              stub server                   JSON committed to repo
responses     Python library    Python      Unit-level API mocking,        No
                                            no server overhead
httpretty     Python library    Python      Socket-level interception,     No
                                            no server needed
WireMock-Java Embedded          Java        In-process JUnit integration,  No
              library                       no Docker needed for Java
```

---

## Default Selection: WireMock (Docker)

WireMock is the default for all tracks because:
1. Language-agnostic — works with Python, Java, JS, Go, C#, any future track
2. Works for both API tests (direct stubs) and UI tests (mocks the backend
   your application calls)
3. Stubs defined as JSON — version-controllable, shareable across team
4. Rich matching: URL, method, headers, request body, regex, JSON path
5. Response templating: dynamic responses using request data
6. Verification: assert stubs were called N times with expected payload
7. Record & Playback: point WireMock at real service to auto-generate stubs

## WireMock Docker Setup

Always use the official WireMock Docker image.

```yaml
# docker-compose.yml (generated at project root)
services:
  wiremock:
    image: wiremock/wiremock:3.3.1
    container_name: wiremock
    ports:
      - "8080:8080"
    volumes:
      - ./mocks/stubs:/home/wiremock/mappings
    command: --global-response-templating --verbose
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/__admin/health"]
      interval: 5s
      timeout: 3s
      retries: 10
```

Start/stop locally:
```bash
docker-compose up -d wiremock    # start
docker-compose down              # stop
```

## Stub File Format (JSON Mappings)

Stubs live in `mocks/stubs/` and are loaded by WireMock at startup.
Each file can contain one or more stub mappings.

```json
// mocks/stubs/user_stubs.json
{
  "mappings": [
    {
      "name": "create-user-success",
      "request": {
        "method": "POST",
        "url": "/api/v1/users",
        "headers": {
          "Content-Type": { "equalTo": "application/json" }
        }
      },
      "response": {
        "status": 201,
        "headers": { "Content-Type": "application/json" },
        "jsonBody": {
          "id": 1001,
          "username": "{{jsonPath request.body '$.username'}}",
          "email": "{{jsonPath request.body '$.email'}}",
          "role": "viewer",
          "created_at": "{{now format='yyyy-MM-dd\\'T\\'HH:mm:ss\\'Z\\''}}"
        }
      }
    },
    {
      "name": "get-user-success",
      "request": {
        "method": "GET",
        "urlPattern": "/api/v1/users/([0-9]+)"
      },
      "response": {
        "status": 200,
        "headers": { "Content-Type": "application/json" },
        "jsonBody": {
          "id": 1001,
          "username": "testuser",
          "email": "test@example.com",
          "role": "viewer",
          "created_at": "2024-01-15T10:00:00Z"
        }
      }
    },
    {
      "name": "get-user-not-found",
      "request": {
        "method": "GET",
        "url": "/api/v1/users/9999"
      },
      "response": {
        "status": 404,
        "headers": { "Content-Type": "application/json" },
        "jsonBody": {
          "error": "User not found",
          "code": "USER_NOT_FOUND"
        }
      }
    }
  ]
}
```

## WireMock Lifecycle Module

Generated at `mocks/wiremock_lifecycle.{ext}` for each track.

**Python version** (generated by Python track):
```python
# mocks/wiremock_lifecycle.py
import requests

WIREMOCK_ADMIN = "http://localhost:8080/__admin"


def register_stub(stub: dict) -> None:
    """Register a stub at runtime (in addition to file-based stubs)."""
    resp = requests.post(f"{WIREMOCK_ADMIN}/mappings", json=stub)
    resp.raise_for_status()


def reset_stubs() -> None:
    """Remove all runtime-registered stubs. File-based stubs remain."""
    requests.post(f"{WIREMOCK_ADMIN}/mappings/reset")


def reset_requests() -> None:
    """Clear the request journal (call count history)."""
    requests.delete(f"{WIREMOCK_ADMIN}/requests")


def verify_called(url_pattern: str, times: int = 1) -> None:
    """Assert a URL was called exactly N times."""
    resp = requests.post(
        f"{WIREMOCK_ADMIN}/requests/count",
        json={"urlPattern": url_pattern}
    )
    actual = resp.json()["count"]
    assert actual == times, (
        f"Expected {url_pattern} to be called {times} time(s), "
        f"but was called {actual} time(s)"
    )


def verify_called_with_body(url: str, body_pattern: dict) -> None:
    """Assert a URL was called with a specific request body."""
    resp = requests.post(
        f"{WIREMOCK_ADMIN}/requests/find",
        json={"url": url, "bodyPatterns": [{"matchesJsonPath": body_pattern}]}
    )
    assert len(resp.json()["requests"]) > 0, (
        f"No request to {url} matched body pattern {body_pattern}"
    )
```

**Java version** (generated by Java track):
```java
// mocks/WireMockLifecycle.java
public class WireMockLifecycle {
    private static final String ADMIN = "http://localhost:8080/__admin";

    public static void registerStub(Map<String, Object> stub) {
        // POST to ADMIN/mappings with stub as JSON body
    }

    public static void resetStubs() {
        // POST to ADMIN/mappings/reset
    }

    public static void verifyCalled(String urlPattern, int times) {
        // POST to ADMIN/requests/count, assert count == times
    }
}
```

## WireMock for UI Backend Mocking

When test_type includes UI, WireMock mocks the backend that the browser
application calls — not the browser itself.

```
Browser → Application (AUT) → WireMock (instead of real backend API)
```

**Environment variable setup**:
```bash
# .env (local)
AUT_BASE_URL=http://localhost:3000      # your app running locally
BACKEND_URL=http://localhost:8080       # WireMock (mocking the real API)
TEST_MODE=mock
```

**Application config** (note: this is the APPLICATION's config, not test config):
The application must be started with `API_BASE_URL=http://localhost:8080`
so it calls WireMock instead of the real backend.

In CI, start WireMock as a service, start the application pointing at
WireMock, then run UI tests.

```yaml
# .github/workflows/ui-tests.yml (CI pattern)
services:
  wiremock:
    image: wiremock/wiremock:3.3.1
    ports:
      - 8080:8080
  app:
    image: my-app:latest
    ports:
      - 3000:3000
    environment:
      API_BASE_URL: http://wiremock:8080   # app calls wiremock, not real API
```

## Alternative Mock Tools

Use these instead of WireMock only if the user explicitly requests them
or if the use case requires it.

### Prism (OpenAPI-driven)

Best when: team has an OpenAPI/Swagger spec and wants zero stub maintenance.
Prism auto-generates stubs from the spec and validates requests against it.

```bash
# Install
npm install -g @stoplight/prism-cli

# Start (auto-stubs from spec)
prism mock ./swagger.json --port 4010

# Or Docker
docker run --init --rm -it \
  -v $(pwd):/tmp \
  stoplight/prism:4 mock /tmp/swagger.json
```

Set base_url = http://localhost:4010 in .env when using Prism.

Trade-off: Prism validates request format (good) but returns example
values from the spec (less control than WireMock for edge cases).

### Mockoon

Best when: team wants a GUI for managing stubs, and stubs committed to
repo as exportable JSON.

```bash
npm install -g @mockoon/cli
mockoon-cli start --data ./mocks/mockoon-environment.json --port 3001
```

Export the Mockoon environment file from the GUI and commit to `mocks/`.

### Language-Native (Python only, unit level)

Use `responses` library for Python when:
- Tests are unit-level (not integration)
- No real service call should ever escape the test process
- Docker overhead is not acceptable

```python
import responses
import requests

@responses.activate
def test_create_user_returns_201():
    responses.add(
        responses.POST,
        "https://api.example.com/users",
        json={"id": 1, "username": "test"},
        status=201,
    )
    # test code here — requests library is intercepted
```

Not recommended as primary mock strategy because it is Python-specific
and breaks the cross-track consistency contract.
